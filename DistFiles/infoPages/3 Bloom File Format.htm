Bloom Format

The following information is for advanced primer or shell developers. If you're just making shells for other people to use, you probably don't need to know any of the following.

Bloom is in experimental mode, and the Bloom format is still evolving in response to needs we discover. Take that into account before converting hundreds of books. The wise thing at this point it to convert one book of each type, so that we can find any missing file-format elements that you will need. The plan is to cease any breaking changes by early 2012.

Bloom documents are XHTML with certain conventions and pre-defined class names, as described below. Of course, you'll want to open up a bloom html file and see for yourself how it is structured.  Two simple examples are the A5Portrait/templatePages.htm, and the Vaccinations.htm, which comes in the Sample Shells project. You can look at the originals or create a book using them, and look at what Bloom does.

Structure

Each page is indicated by <div class="-bloom-page">. The only things that the user can edit are images and <textarea>s.  There can be multiple <textarea>s together inside a <p>, each marked with a distinct lang attribute. Bloom will take care of only showing the user the correct <textarea> which matches the language the user is currently working in.

Formatting

Formatting within these pages is typically done by adding a unique class to the div for a particular page layout. That class then triggers the CSS style rules you've made for the page.  Using Stylizer, this can be much less painful that it sounds.  It bears repeating: all of this is only for making new page templates, or for jobs (like fancy literacy primer templates) that can't use the standard page templates found in the A5Portrait Template. 

StyleSheets

  • papersize/orientation. E.g. a5portrait.css. Currently, this is the only size/portrait available.

    • The next three stylesheets are part of Bloom and are not available for customization under any circumstances.

  • basePage.css: always applied. Used for styles which always apply, regardless of mode or paper size.
  • editMode.css is applied during editing
  • previewMode.css is applied during viewing and printing

    • Note: when a new book is created, each of these pages is copied in to the folder with the book, so that if it is moved to a computer which doesn't have bloom, it will be still viewable.  However, each time the book is opened, Bloom will replace the stylesheets in that folder with the versions in the Bloom program directory.  So if you're working on one of the standard stylesheets, be careful to not let Bloom over-write your work.

    Built-in Classes

    While most classes are just there to link with the CSS stylesheet, Bloom has several classes which the c# code looks for and uses:

  • -bloom-page: Each page is indicated by a <div> which has this class.
  • -bloom-vernacularBookTitle: This is shown on the cover, sets the <title> element of the html, sets the name of the book's folder, and finally, the book's html file.
  • -bloom-topicInNationalLanguage: e.g. "Health", "Math". Currently only used by stylesheets, but reserved for future c# code use.
  • -bloom-nameOfLanguage: this is automatically filled in, so it should only appear in uneditable elements.
  • -bloom-requiredPage: Page <div>s with this class are always added to a new book, and the user is not allowed to remove them.
  • -bloom-singleton: Only a single copy of this page may be added to a book.
  • -bloom-extraPage: When a book is created, any pages marked with this class will not be initially added to the new book. If the user wants one of these pages, he has to add it explicitly (by clicking on its icon in the list of template pages).
  • -bloom-showNational: indicates that it shouldn't be translated. See the section "other language-related things" below.
  • -bloom-guidOfTemplateBook. See the section below on Lineage.
  • -bloom-pageLineage. See the section below on Lineage.

    • Lineage

    (This area is especially immature)

    While Bloom doesn't yet provide it, we may someday need the ability to update all the pages stemming from a certain template page. For example, an entity might change their standard title page format, and they might like to have a way of updating the existing books that were using the old format. To make that possible in the future, we need to store the lineage now. Here's how it works at the moment:

    <a href="0151a8d0-8718-41f0-a7fb-71c6995fa80a;bfc0737c-f76a-4a34-93ba-d28cb483823a" class='-bloom-pageLineage' style='visibility:hidden'></a>

    Notice that the href above has 2 guids (Globally Unique Identifiers), separated by a semicolon. This tells us that this page came from a page with the id bfc0737c-f76a-4a34-93ba-d28cb483823a, which in turn came from a page with the id 0151a8d0-8718-41f0-a7fb-71c6995fa80a.

    Careful: if that closing tag is missing, everything following this will be invisible.

    "Constants"

    If a <textarea> has a class which begins with an underscore, Bloom will copy the text in that textarea into other paragraphs, spans, and textareas which have the same class. This allows the user to translate something once, and have it show up all over. For example, the following textarea is used in a study book, where many of the pages have a heading which, in English, just says "Questions".

    <textarea lang="en" class="Heading2 _questionsHeading">Questions</textarea>

     

    Shell Books and Languages

    Shell books may come with source texts in multiple languages, so that users in different countries or with different language strengths can use the source language which works best for them.

    So, how do we get all those languages in the same document? This area is still maturing. Currently, the approach is to surround each textarea in a p, then have one <textarea> for each source language. Each textarea must have its lang attibute set to a proper ISO639 (Ethnologue) code:

    <p>
         <textarea lang="tpi"> "Taim yu planim gaden yu save wokim banis.</textarea>
         <textarea lang="en" class="hideMe"> When you plant a garden, you always build a fence.</textarea>
    </p>

    Note: the 'hideMe' is currently used by Bloom to hide all but one language.  In the future, we may instead use a dynamically-created css to turn languages on and off.

    Now, if Bloom makes a book using the above paragraph, it will make a new <textarea> for the vernacular; the resulting book will then have 3 textareas in this paragraph, with all but the vernacular containing the class 'hideMe'.

    That 'hideMe' class is just to keep the source texts out of the page in the browser. In edit mode the Bloom application still reads these <textarea>s, and shows the contents source-text pane, to help the user do his translation.

    Other language-related things

    Books in minority languages normally have some text that is in a language of wider communication. For example, the title of the book is often printed in the national language, below the vernacular one.  To tell Bloom that we don't want to translate a text, we use the -bloom-showNational tag (REVIEW: this currently just means "don't mess with this element"; it doesn't do something smart like set the language to the local national language. This is fine if the shell was designed for this country.)

    <textarea lang="en" class="credits -bloom-showNational" title="Enter the contributions made by writers, illustrators, editors, etc." id="df9d7385-8cfe-43d0-bfc2-6b3423a79954"> </textarea>
